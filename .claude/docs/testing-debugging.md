# Testing, Debugging & Evaluation

## Overview

SAR-env currently lacks formal unit or integration tests. Testing relies on **manual simulation runs** with deterministic task overrides, visual inspection via the Flask GUI, and post-hoc log analysis. Thread safety is verified through design (ThreadPoolExecutor isolation, Lock-protected SharedMemory). Debugging workflows leverage manual plan injection, structured logging, and score validation.

## Testing Infrastructure (None Exists)

**No formal test suite**: The project has no `test_*` files, pytest configuration, or CI integration. Testing is entirely manual through simulation execution.

**Recommended approach** (not implemented): Unit tests for modules (validation, planning, tool registry), integration tests for agent decision cycles, end-to-end tests for multi-agent cooperation scenarios. Current workflow: modify code, run full simulation, inspect logs and GUI.

## Manual Testing via Simulation

**Primary testing method**: Run `python main.py` with specific configurations and observe behavior through logs and visualizer.

### Deterministic Testing with Manual Plans

Set `manual_plans_file = "manual_plans.yaml"` in `main.py` to bypass LLM planning entirely. Hardcode exact task assignments per iteration to create reproducible test scenarios. Useful for:

- **Regression testing**: Verify specific agent behaviors without LLM variability
- **Edge case reproduction**: Force agents into cooperative deadlock, obstacle interactions, victim handoff
- **Performance benchmarking**: Isolate execution speed vs LLM latency

**Configuration** (`manual_plans.yaml`): Define `iterations` section with per-agent tasks and optional plans. Example: force agent 0 into area 1, agent 1 into area 2, check coordination. If iteration count exceeds defined entries, last entry repeats. Plans bypass PlanningModule LLM; tasks bypass EnginePlanner task generation.

**Files**: `manual_plans.yaml` (test scenarios), `engine/engine_planner.py` (`_build_manual_tasks()` loader).

### Configuration-based Testing

**Capability testing**: Set `agent_presets=['scout', 'heavy_lifter']` and `capability_knowledge='discovery'` to verify agents learn restrictions through environment feedback. Set to `'informed'` to test prompt-based capability awareness.

**Planning mode testing**: Toggle `planning_mode='simple'` vs `'dag'` to validate flat task lists vs conditional branching. DAG mode auto-detects "if/else" patterns in task descriptions (`agents1/modules/planning_module.py`).

**Agent count scaling**: Test with `num_rescue_agents=1` (single agent baseline) up to 5 (high coordination complexity). Each agent requires own Ollama instance on consecutive ports.

**World presets**: Use `world_preset='static'|'preset2'|'preset3'|'random'` with fixed `world_seed` for reproducible environments (`worlds1/world_presets.py`).

## Debugging Workflows

### 1. Real-time Visualization

**Flask GUI** (port 3000): Navigate to `http://localhost:3000` after starting simulation. Shows:
- God view: All agents, victims, obstacles, explored areas
- Per-agent views: Individual perspectives with vision range
- Tick counter, score updates, action execution

**Disable for headless**: Set `enable_gui=False` in `main.py` for server/cluster runs (e.g., Delft Blue).

**Implementation**: `SaR_gui/visualization_server.py` (custom fork of `matrx_visualizer/`).

### 2. Log Analysis

**Output locations**: Logs generated in `logs/` directory (auto-created).

**Key log files**:
- `score.json`: Current simulation state (victims_rescued, total_victims, score, block_hit_rate). Updated each iteration. Format: `{"score": 0, "block_hit_rate": 0.0, "victims_rescued": 0, "total_victims": 8}`. Read by EnginePlanner for termination checks (`engine/engine_planner.py`, `_read_score_info()`).
- `iteration_history.json`: Per-iteration summary with task assignments, LLM summaries, scores. Saved on simulation completion (`main.py` post-run). Dataclass: `engine/iteration_data.py`.
- MATRX action logs: Auto-generated by `loggers/ActionLogger.py` in timestamped subdirectories. CSV format with tick number, agent actions, locations, score, completeness.

**Logging setup**: MATRX framework handles automatic action logging. Custom loggers extend `GridWorldLogger` (`loggers/ActionLogger.py`).

### 3. Thread Safety Verification

**Design-based safety**: No formal tests, but architecture ensures correctness:
- **LLM calls**: Each agent submits to shared ThreadPoolExecutor (8 workers, `agents1/async_model_prompting.py`). Non-blocking Futures prevent tick loop blocking. EnginePlanner uses separate pool (4 workers, `engine/engine_planner.py`).
- **SharedMemory**: All cross-agent coordination (rendezvous for cooperative carry, task assignments) protected by `threading.Lock` (`memory/shared_memory.py`).
- **Action validation**: Performed synchronously per agent before dispatch, no race conditions (`helpers/logic_module.py`).

**Verification approach**: Run high-concurrency scenarios (5 agents, DAG mode, heavy cooperation) and check logs for state corruption or action conflicts. No formal concurrency tests exist.

### 4. LLM Timeout & Retry Handling

**Retry decorator**: `@_retry_with_backoff(retries=5, base_wait_time=1)` on `_completion_dispatch()` (`agents1/async_model_prompting.py`). Exponential backoff: 1s, 2s, 4s, 8s, 16s. After 5 failures, exception propagates to agent.

**Timeout**: HTTP requests to Ollama use 120-second timeout (`timeout=120` in `_completion_requests()`). Ollama SDK backend has no explicit timeout (relies on model generation limits).

**Agent-level handling**: SearchRescueAgent catches exceptions in `get_llm_result()` call (`agents1/agent_sar.py`). Falls back to Idle action on failure, logs warning. No retry at agent level — relies on decorator.

**EnginePlanner handling**: Task generation failures logged, iteration continues with empty tasks. Summarization failures use fallback summary. No hard stop on LLM errors.

### 5. Action Validation Debugging

**Pre-dispatch validation**: `ActionValidator` checks all LLM-chosen actions before MATRX execution (`helpers/logic_module.py`). Catches:
- Out-of-bounds moves
- Invalid object IDs (victim/obstacle doesn't exist)
- Wrong action for object type (e.g., RemoveObject on victim)
- Missing teammates for cooperative actions
- Capability violations (if `capability_knowledge='informed'`)

**Validation bypass**: In `'discovery'` mode, capability checks disabled to let environment teach agents through failure.

**Feedback loop**: ValidationResult contains LLM-friendly error message fed back into agent's next reasoning cycle via TaskCritic (`agents1/modules/task_critic_module.py`).

**Files**: `helpers/logic_module.py` (ActionValidator), `agents1/llm_agent_base.py` (validation integration).

### 6. Reproducing Issues

**Steps to isolate bugs**:
1. **Simplify**: Reduce to `num_rescue_agents=1`, `planning_mode='simple'`, single area
2. **Fix randomness**: Set `world_seed=42`, use `manual_plans_file` for deterministic tasks
3. **Enable verbose logging**: Add debug prints in `agents1/agent_sar.py` (no logging level config exists)
4. **Inspect state**: Use Flask GUI to watch agent positions, memory state
5. **Check validation**: Add breakpoints in ActionValidator to catch invalid LLM outputs
6. **LLM inspection**: Log raw LLM responses in `async_model_prompting.py` (`_completion_dispatch()` return value)

## Evaluation Metrics

### Score Validation

**Scoring system** (`score.json`):
- `score`: Points earned (critically injured = 6, mildly injured = 3)
- `victims_rescued`: Count of victims delivered to drop zone
- `total_victims`: Total victims in world (default 8)
- `block_hit_rate`: Ratio rescued/total (1.0 = mission complete)

**Termination**: EnginePlanner checks `block_hit_rate >= 1.0` or `iteration >= max_iterations` (`engine/engine_planner.py`, `decide_next_step()`).

**Post-run analysis**: `loggers/OutputLogger.py` parses MATRX action logs to compute completeness, action counts, trust metrics (legacy feature). Saves `output.csv` in log directory.

### Performance Metrics (Manual Extraction)

**From iteration_history.json**:
- Iterations to completion
- Task assignments per iteration
- LLM summary quality (subjective)

**From MATRX logs**:
- Total ticks (1 tick = 0.1s)
- Unique agent actions count
- Cooperative action frequency (CarryObjectTogether, RemoveObjectTogether)

**No automated benchmarking**: Current workflow requires manual spreadsheet extraction and comparison across runs.

## Known Debugging Pitfalls

**LLM variability**: Same configuration produces different outcomes due to model non-determinism. Use `manual_plans_file` for reproducibility.

**Cooperative deadlock**: Two agents stuck waiting for each other due to poor task coordination. Check SharedMemory rendezvous state (`memory/shared_memory.py`, `register_rendezvous()` calls).

**Action validation false positives**: Validator may reject valid actions if world state is stale. Ensure state passed to `ActionValidator.validate()` is current tick's state.

**ThreadPoolExecutor starvation**: If all workers blocked on slow LLM calls, agents stall. Monitor worker count vs active agents (default 8 workers supports 5+ agents).

**Missing Ollama instances**: Each agent needs own Ollama on port `11434+N`. Forgotten instances cause connection errors, agent falls back to Idle.

- **engine/engine_planner.py**: Iteration loop, termination checks, score reading
- **agents1/agent_sar.py**: LLM call exception handling, Idle fallback
- **agents1/async_model_prompting.py**: Retry decorator, timeout config, ThreadPoolExecutor
- **helpers/logic_module.py**: ActionValidator (pre-dispatch checks)
- **memory/shared_memory.py**: Lock-protected cross-agent state
- **loggers/**: ActionLogger (MATRX action logging), OutputLogger (post-run analysis)

- **logs/score.json**: Real-time score/progress (read by planner)
